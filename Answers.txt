Q1. Explain the difference between REST and GraphQL APIs.
A: REST (Representational State Transfer) is an architectural style where each endpoint represents a resource (e.g., /users, /posts). Each endpoint returns a fixed data structure. GraphQL, on the other hand, is a query language for APIs where the client specifies exactly what data it needs in a single request. REST can lead to over-fetching (getting more data than needed) or under-fetching (needing multiple calls), while GraphQL solves both by letting clients request only the fields they need. REST uses multiple endpoints; GraphQL uses a single endpoint. REST is simpler and widely adopted; GraphQL is more flexible and efficient for complex, nested data needs like dashboards or mobile apps.

Q2. What is the difference between SQL and NoSQL databases? When would you use each?
A: SQL databases are relational, use structured schemas, and store data in tables with rows and columns. They support ACID transactions and are great for structured data with complex relationships — like banking systems or e-commerce platforms. Examples: PostgreSQL, MySQL. NoSQL databases are non-relational and support flexible schemas. They come in types like document (MongoDB), key-value (Redis), column (Cassandra), and graph (Neo4j). They are ideal for unstructured or semi-structured data, high-speed reads/writes, and horizontal scaling — like real-time analytics, social feeds, or IoT data. I would use SQL when data integrity and relationships matter, and NoSQL when scalability and flexibility are the priority.

Q3. Explain how async/await works under the hood in JavaScript.
A: async/await is syntactic sugar over Promises. When you mark a function as async, it automatically returns a Promise. The await keyword pauses execution of that function until the awaited Promise resolves or rejects — but it does not block the main thread. Under the hood, JavaScript's event loop handles this: when await is hit, the function is suspended and control returns to the call stack. Once the Promise settles, the microtask queue picks up the continuation and resumes the function. This makes asynchronous code look synchronous and easier to read, without the callback hell of older patterns. Error handling is done using try/catch, which wraps rejected Promises cleanly.

Q4. What is a closure in programming? Give a real-world example.
A: A closure is a function that retains access to its outer (enclosing) scope even after the outer function has returned. In JavaScript, every function forms a closure over the variables in its lexical scope. A real-world example is a counter function: you define a function makeCounter() that initializes count = 0 and returns an inner function that increments and returns count. Each call to makeCounter() creates an independent counter because the inner function closes over its own count variable. Closures are heavily used in event handlers, factory functions, and module patterns to encapsulate private state without polluting the global scope.

Q5. How does indexing improve database query performance?
A: Without an index, a database performs a full table scan — reading every row to find matches, which is O(n). An index creates a separate data structure (usually a B-tree or hash) that maps column values to their row locations, allowing the database to find rows in O(log n) time. This dramatically speeds up SELECT queries with WHERE, JOIN, or ORDER BY clauses on indexed columns. However, indexes come with trade-offs: they consume extra storage and slow down INSERT, UPDATE, and DELETE operations because the index must be updated too. The best practice is to index columns that are frequently queried or used in joins, and avoid over-indexing write-heavy tables.

Q6. What is the difference between authentication and authorization?
A: Authentication is the process of verifying who a user is — confirming their identity. This is done through credentials like username/password, OTP, biometrics, or OAuth tokens. Authorization is the process of verifying what a user is allowed to do — determining their permissions after identity is confirmed. For example, both a regular user and an admin may be authenticated (logged in), but only the admin is authorized to delete records. In practice, authentication comes first; authorization follows. Tools like JWT handle authentication by encoding identity in tokens, while role-based access control (RBAC) or attribute-based access control (ABAC) handle authorization logic.

Q8. What are React hooks and why were they introduced?
A: React hooks are functions that let you use state and other React features in functional components without writing class components. They were introduced in React 16.8 to address several problems with class components: complex lifecycle methods were hard to understand, stateful logic was difficult to reuse across components, and classes confused both humans and machines (with issues like binding 'this'). Key hooks include useState for local state, useEffect for side effects like data fetching and subscriptions, useContext for consuming context, and useRef for mutable values. Custom hooks allow you to extract and share stateful logic across components in a clean, composable way.

Q9. How would you optimize a slow-loading web page?
A: I would approach this in layers. On the network side: enable compression (gzip/brotli), use a CDN, reduce DNS lookups, and enable HTTP/2. On the asset side: minify and bundle JS/CSS, lazy-load images and non-critical scripts, use next-gen image formats (WebP), and implement code splitting so only the needed JS loads per route. On the rendering side: reduce render-blocking resources, defer non-critical scripts, and use server-side rendering or static generation for faster first paint. On the data side: optimize API calls, implement caching (Redis, browser cache), and paginate large data sets. I would use tools like Lighthouse, WebPageTest, and Chrome DevTools to identify bottlenecks before and after changes.

Q10. What is the CAP theorem and how does it affect distributed systems?
A: The CAP theorem states that a distributed system can only guarantee two of three properties simultaneously: Consistency (every read returns the most recent write), Availability (every request gets a response, even if not the latest data), and Partition Tolerance (the system continues operating despite network partitions). Since network partitions are unavoidable in distributed systems, designers must choose between CP (consistent but may be unavailable during partitions — e.g., HBase, Zookeeper) or AP (available but may return stale data — e.g., Cassandra, DynamoDB). Understanding CAP helps architects make informed trade-offs. For example, a banking app prioritizes CP for data accuracy, while a social media feed may prefer AP for uptime and speed.

Q19. Describe a bug you spent the most time debugging. How did you solve it?
A: The hardest bug I debugged was a race condition in a real-time app. Occasionally, two users editing the same record simultaneously would result in one update silently overwriting the other. The bug was intermittent and only appeared under concurrent load, making it nearly impossible to reproduce locally. I added detailed server-side logging with timestamps and user IDs, then wrote a script to simulate concurrent requests. The logs revealed that both requests read the same record version before either wrote back. I solved it by implementing optimistic locking — adding a version field to the record and rejecting updates where the version didn't match the current database value. The fix was three lines of code after two days of investigation.
